# DatsSol: нормализованные правила и инструкция для LLM-агента

Этот документ предназначен для других LLM, которые должны принимать решения в игре DatsSol.
Он переформулирует исходные правила в более машинно-прикладной вид.

## 1. Источник истины и приоритеты

1. Главный источник истины: фактическая логика сервера.
2. Если документация и наблюдаемое поведение расходятся, действуй по серверу.
3. Организаторы могут менять баланс и отдельные механики; проверяй актуальные данные через API на каждом ходу.
4. Ключевые динамические параметры (длительность хода, длина раунда в тестах, финальные настройки) могут изменяться.

## 2. Базовая модель игры

1. Карта: плоскость XY.
2. Клетка бывает двух типов:
    - пустыня: можно терраформировать и строить;
    - горы: нельзя терраформировать/строить.
3. Ход: базовый тик игры (обычно 1 секунда, может быть изменено).
4. Раунд: обычно 600 ходов (в тестах может отличаться).
5. Игрок стартует с одной плантации, которая является ЦУ (центр управления), появляется при первом запросе.

## 3. Порядок обработки сервером в конце хода

Сервер выполняет этапы строго в таком порядке:

1. Апгрейды плантаций.
2. Ремонт/стройка.
3. Диверсии.
4. Атака по логову бобров.
5. Перенос ЦУ.
6. Атака бобров.
7. Деградация потерянных плантаций.
8. Урон по недостроенным плантациям (если 1 ход без прогресса).
9. Терраформация и начисление очков.
10. Респавн игроков.
11. Природные катаклизмы.

Важно:
1. На каждом этапе сервер проходит по всем юнитам.
2. Юнит, уничтоженный на текущем этапе, не участвует в следующих этапах этого же хода.

## 4. Плантации: параметры и ограничения

Базовые параметры (по умолчанию):

1. `HP`: текущее здоровье.
2. `MHP`: максимум HP = 50.
3. `TS`: скорость терраформации = 5 (% за ход).
4. `CS`: скорость строительства = 5.
5. `RS`: скорость ремонта = 5.
6. `SE`: эффективность диверсии = 5.
7. `BE`: урон по логову бобров = 5.
8. `DS`: скорость деградации = 10.
9. `AR`: action range = 2.
10. `SR`: signal range = 3.
11. `VR`: vision range = 3.

Лимит плантаций:
1. Базово: 30.
2. Потерянные (изолированные) плантации тоже занимают лимит.

## 5. Геометрия радиусов и связность сети

### 5.1 Радиусы AR/SR/VR

Радиус определяется по каждой оси отдельно:

- цель достижима, если одновременно:
    - `abs(dx) <= R`
    - `abs(dy) <= R`

Это квадратная область (метрика Чебышёва), не круг.

### 5.2 Связность с ЦУ

Управляемыми считаются только плантации, соединенные с ЦУ цепочкой по 4-соседству (вверх/вниз/влево/вправо), диагональ не считается.

## 6. ЦУ и критические последствия

1. ЦУ можно перенести на соседнюю плантацию.
2. При переносе HP существующих плантаций не меняется.
3. Если ЦУ уничтожено/потеряно:
    - все плантации игрока уничтожаются;
    - игрок респавнится;
    - штраф по очкам: `max(1, floor(current_score * 0.05))`;
    - уже купленные апгрейды сохраняются.

## 7. Потерянные (изолированные) плантации

Если разорвалась связь с ЦУ:

1. Плантация неуправляема.
2. Сбрасывает/прекращает ранее выданную команду.
3. Продолжает терраформировать клетку, но без начисления очков игроку.
4. Теряет HP со скоростью `DS` за ход.

Если связь восстановлена, плантация снова работает как обычная.

## 8. Действия плантаций за ход

Одна плантация за ход может:

1. Всегда терраформировать клетку под собой (автоматически, без команды).
2. Дополнительно выполнить одно активное действие:
    - строительство;
    - ремонт;
    - диверсия;
    - атака логова бобров.

## 9. Маршрут команды и потеря эффективности

Формат маршрута действия:

1. Первая координата: автор команды (своя плантация).
2. Вторая: выходная точка (своя плантация, может совпадать с автором).
3. Третья: цель.

Ограничения:

1. Выходная точка должна быть достижима по `SR`.
2. Цель должна быть достижима от выходной точки по `AR`.
3. Для всех команд, использующих одну и ту же выходную точку в одном ходу, действует штраф эффективности:
    - первая команда через эту точку: без штрафа;
    - каждая следующая: `CS/RS/SE/BE` уменьшается на 1;
    - минимум 0.

Практическая формула:
`effective_stat = max(base_stat - reuse_index, 0)`,
где `reuse_index` = 0 для первой команды через выходную точку, 1 для второй и т.д.

## 10. Строительство новой плантации

1. Строить можно на клетке в пределах `AR` от выходной точки.
2. В строительство одной клетки могут вкладываться несколько плантаций (с учетом штрафа эффективности по выходной точке).
3. Недостройка - отдельный юнит:
    - не получает бонусы от апгрейда `max_hp`;
    - превращается в плантацию при достижении 50 HP прогресса;
    - после завершения получает `HP = MHP` текущего уровня;
    - после завершения 3 хода иммунитета к:
        - действиям соперников;
        - атакам бобров;
        - природным катаклизмам.

Конкуренция за одну клетку:

1. Каждый игрок ведет свой прогресс стройки отдельно.
2. Если один завершил раньше других, на клетке остается его плантация.
3. Если несколько игроков завершили в один ход, прогресс обнуляется у всех (строить заново).

Превышение лимита плантаций:

1. Если начато строительство сверх лимита, то в момент первого прогресса новой стройки исчезает самая старая завершенная плантация игрока.
2. Если несколько самых старых одного возраста, удаляется случайная.
3. Можно случайно потерять ЦУ, если оно окажется «самым старым».

Деградация недостроя:
1. Если у стройки нет прогресса 1 ход, она начинает деградировать со скоростью `DS`.

## 11. Ремонт

1. Можно чинить только свои плантации.
2. Восстановление HP за ход = эффективное `RS`.
3. Плантация не может ремонтировать сама себя (нужна другая как исполнитель).

## 12. Диверсия

1. Диверсия по чужой плантации снимает HP на эффективное `SE` за ход.
2. При разрушении цели диверсант получает очки, эквивалентные ценности клетки.
3. Если по цели били несколько игроков:
    - побеждает тот, кто в последний ход нанес больше урона;
    - при равенстве урона очки делятся поровну.
4. Текущий прогресс терраформации клетки сохраняется после разрушения плантации.
5. Нельзя атаковать недострой соперника как диверсией:
    - вместо этого на этой клетке работает механика конкурирующего строительства.

## 13. Терраформация и очки

1. Клетка стартует с 0% терраформации.
2. Каждый ход плантация добавляет к прогрессу `TS` процентов.
3. Начисление за 1%:
    - обычная клетка: 10 очков (максимум 1000 за 100%);
    - усиленная клетка (`x % 7 == 0` и `y % 7 == 0`): 15 очков (максимум 1500).
4. При достижении 100%:
    - клетка считается завершенной;
    - плантация на этой клетке исчезает.
5. Через 40 ходов после полной терраформации клетка начинает деградировать:
    - минус 10% прогресса в ход.
    - параметр в тестах может быть изменен.

## 14. Логово бобров

1. Логово имеет 100 HP.
2. Регенерация логова: +5 HP/ход.
3. Логово атакует все плантации в радиусе `AR=2`.
4. Урон бобров: 15 HP/ход.
5. Урон бобров применяется и к недостроенным плантациям.
6. За уничтожение логова дается награда = `10x` ценности клетки.
7. Если в добивании участвуют несколько игроков:
    - награду получает нанесший максимум урона в последний ход;
    - при равенстве награда делится поровну.
8. После уничтожения логова клетка доступна для терраформации.

## 15. Природные катаклизмы

### 15.1 Песчаная буря

1. Возникает с вероятностью 10% на ход.
2. На карте одновременно может быть только одна буря.
3. Формируется 5 ходов; в фазе формирования урон не наносит.
4. После формирования движется по диагонали, проходя через центр карты.
5. Скорость движения выбирается случайно при появлении: 5..15 клеток/ход.
6. Урон бури: 2 HP/ход плантациям в области.
7. Буря не может уничтожить плантацию: минимальный HP от бури = 1.

### 15.2 Землетрясение

1. Вероятность 5% на ход.
2. Мгновенный урон всем плантациям и стройкам: 10 HP.
3. На бобров катаклизмы не влияют.
4. Если плантация уничтожена землетрясением, очки за прежние диверсии по ней не выдаются.

## 16. Апгрейды

Механика:

1. Каждые 30 ходов начисляется 1 очко апгрейда.
2. Максимум накопления очков: 15.
3. За ход можно отправить покупку одного апгрейда (если есть очки).

Ветки и лимиты:

1. `repair_power`: `RS +1`, максимум 3.
2. `max_hp`: `MHP +10`, максимум 5.
3. `settlement_limit`: лимит плантаций `+1`, максимум 10.
4. `signal_range`: `SR +1`, лимит в документе не указан.
5. `decay_mitigation`: уменьшение `DS` на 2, максимум 3, работает и на стройки.
6. `earthquake_mitigation`: уменьшение урона землетрясения на 2, максимум 3, работает и на стройки.
7. `beaver_damage_mitigation`: уменьшение урона от бобров на 2, максимум 5, работает и на стройки.
8. `vision_range`: `VR +2`, максимум 5.

## 17. Очки, баллы и победитель

### 17.1 Очки в раунде

Очки раунда дают:
1. Терраформация.
2. Диверсии (разрушение чужих плантаций).
3. Уничтожение логова бобров.

### 17.2 Баллы за места

1. Команды ранжируются по очкам раунда.
2. За место начисляются турнирные баллы по таблице.
3. Если в конце раунда у команды очки > 0, гарантирован минимум 1 балл.
4. При равенстве очков команды делят одно место.

Ключевые значения таблицы:
1. 1 место: 150
2. 2: 114
3. 89
4. 69
5. 58
6. 50
7. 44
   8..12: 43, 42, 41, 40, 39
   13..50: убывание до 1
   51+ место: 1

### 17.3 Глобальный победитель (сумма по всем финальным раундам)

Приоритет tie-break:
1. Больше баллов.
2. Меньше потерянных плантаций.
3. Больше уничтоженных бобров.
4. Больше диверсий.

## 18. Критерий «активного участника»

Команда активна, если в как минимум половине финальных раундов терраформировала не менее 2 клеток за раунд.

## 19. API для LLM-агента

### 19.1 Авторизация

Во все запросы передавать заголовок:
`X-Auth-Token: <token>`

### 19.2 GET `/api/arena`

Назначение: состояние мира на текущий ход.

Ключевые поля ответа:

1. `turnNo`, `nextTurnIn`.
2. `size`.
3. `actionRange`.
4. `plantations[]`:
    - `id`, `position`, `isMain`, `isIsolated`, `immunityUntilTurn`, `hp`.
5. `enemy[]`: `id`, `position`, `hp`.
6. `mountains[]`.
7. `cells[]`: `position`, `terraformationProgress`, `turnsUntilDegradation`.
8. `construction[]`: `position`, `progress`.
9. `beavers[]`: `id`, `position`, `hp`.
10. `plantationUpgrades`:
    - `points`, `intervalTurns`, `turnsUntilPoints`, `maxPoints`, `tiers[]`.
11. `meteoForecasts[]`:
    - `kind`, `turnsUntil`, `id`, `forming`, `position`, `nextPosition`, `radius`.

### 19.3 POST `/api/command`

Назначение: отправка команд на текущий ход.

Тело:

```json
{
  "command": [
    {
      "path": [[authorX, authorY], [outputX, outputY], [targetX, targetY]]
    }
  ],
  "plantationUpgrade": "repair_power",
  "relocateMain": [[fromX, fromY], [toX, toY]]
}
```

Тип действия определяется целью (третья точка `path`):
1. Своя плантация -> ремонт.
2. Чужая плантация -> диверсия.
3. Логово бобров -> атака логова.
4. Пустая клетка -> строительство.

Важная проверка:
1. В ходе должно быть минимум одно полезное действие:
    - хотя бы один `command`, или
    - `plantationUpgrade`, или
    - `relocateMain`.
2. Иначе сервер может вернуть ошибку про `empty command`.

Типичные `errors`:
1. `command already submitted this turn`
2. `empty command: no plantation actions, no relocateMain, and no plantationUpgrade provided`

### 19.4 GET `/api/logs`

Возвращает игровые логи игрока:
1. `time`
2. `message`

Если игрок еще не зарегистрирован, возможен ответ:
`code = 3` и ошибка `you are not registered in the game, please register first`.

## 20. Рекомендованный цикл принятия решения для LLM

На каждом ходу:

1. Прочитать `GET /api/arena`.
2. Обновить внутреннее состояние:
    - связность с ЦУ;
    - лимит плантаций;
    - иммунитеты;
    - прогноз катаклизмов;
    - доступные апгрейды.
3. Сгенерировать кандидаты действий с проверкой:
    - валидность по SR/AR;
    - риск потери ЦУ;
    - штраф эффективности на выходных точках;
    - локальные угрозы (бобры, буря, землетрясение).
4. Отсортировать приоритеты:
    - защита ЦУ и сохранение связности;
    - конверсия в очки (усиленные клетки, добивания бобров, выгодные диверсии);
    - расширение сети без опасного переполнения лимита.
5. Опционально добавить `plantationUpgrade`/`relocateMain`.
6. Отправить `POST /api/command`.
7. Обработать `errors`, залогировать причины, скорректировать стратегию на следующий ход.

## 21. Частые ошибки, которых нужно избегать

1. Оценивать радиус как круг вместо квадрата по осям.
2. Игнорировать штраф эффективности при множестве команд через одну выходную точку.
3. Забывать, что изоляция от ЦУ отключает управление и очки с терраформации.
4. Пытаться чинить плантацию самой собой.
5. Не учитывать риск удаления самой старой плантации при стройке сверх лимита (включая риск потери ЦУ).
6. Бить недострой диверсией вместо корректной конкурирующей стройки.
7. Отправлять пустой ход без `command`/`plantationUpgrade`/`relocateMain`.
